OOK = On-Off Keying

  • 1 = 433 MHz carrier wave transmitted
  • 0 = Radio silence (no transmission)

  • Varies signal strength (amplitude)

What's inside a 433 MHz signal?

┌─────────────┬────────────────────┬─────────────┬─────────────────┐ │ PREAMBLE │ ADDRESS / ID │ COMMAND │ REPEATED │ │ (sync) │ (which device) │ (which btn) │ 5–20 times │ └─────────────┴────────────────────┴─────────────┴─────────────────┘ 101010... 01011010 1100 0001 → → → → →
  • Preamble - alternating 1010... wakes up the receiver, helps it calibrate
  • Address - which device? Set by DIP switches or burned in at factory
  • Command - which button? (button A, B, C…)
  • Repetition - same packet sent 5–20× per button press for reliability
💡 More repetitions = more chances to capture the signal!
Wireless Fundamentals

Fixed Code vs Rolling Code

❌ Fixed Code (PT2262, EV1527)
  • Same code every single press
  • Like a password you shout out loud
  • Capture once → replay forever
  • Used in: doorbells, cheap plugs, old garages
✓ Rolling Code (KeeLoq/HCS301)
  • Code changes every press (PRNG)
  • Like a one-time password (OTP)
  • Replayed codes are rejected
  • Used in: modern car fobs, Security+ garage doors
Quick quiz: Your TV remote vs. your car key - which is more secure?

Answer: Car key (rolling code). Many garage doors, smart plugs, doorbells? Fixed code. 😬

The Vulnerability

Why it works - four missing things

Missing Feature What It Means
❌ No Encryption Bits fly through the air in cleartext - anyone can read them
❌ No Timestamps Receiver can't tell if signal is "fresh" or recorded months ago
❌ No Authentication No proof of who sent it - attacker looks identical to remote
❌ No Sequence Nums Same code is valid forever - replaying 1000× still works
Key Point

Fixed-code chips such as EV1527 and PT2262 are replayable because they lack freshness checks, authentication, and state.

Secure Alternatives

So how do we fix this?

Solution How It Works Strength
Rolling Codes (KeeLoq) Code changes every press using shared PRNG seed Strong
Challenge-Response Device sends random challenge; remote computes response Strong
AES-128 Encryption Encrypted payload - captured ciphertext is useless Very Strong
Timestamps Signals expire after N seconds - replays rejected Moderate
BLE / Z-Wave / Matter Full protocol stack with pairing, encryption, auth Very Strong

#4827
Rejected - that code was already used once.
  • Remote and receiver share the same secret state.
  • Each button press advances to a new valid code.
  • Replaying an old code fails because the receiver moved on.
Secure Alternatives

What should YOU do?

As a user:

  • Check your garage door: look for Security+ or Intellicode
  • Replace old fixed-code smart plugs with Matter/Z-Wave
  • Don't buy the cheapest RF device - security costs a bit more
  • Treat 433 MHz like an unlocked door - don't rely on it for security

As a developer/engineer:

  • Use rolling codes at minimum (HCS301, KeeLoq)
  • Use AES encryption for sensitive applications
  • Consider BLE or Thread - security is built in
  • Threat model first: who can access the signal?
